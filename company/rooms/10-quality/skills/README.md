# Room 10-quality — Skills

> `10-quality` owns Gate 5 outright — the only gate this room owns — so its `/sofi-gate` use is the real owner-room exit decision, not a scoped contribution check like most other rooms run. It also sits at the far end of the token-economy loop: everything upstream (design, architecture, build) either paid off in a clean Gate-5 pass or it didn't, and this room is where that bill gets read. This is the room's own reading of when each `/sofi-*` skill fires, not a duplicate of the skill files themselves (`.claude/skills/sofi-*/`).

## Skills this room wields, and when

| Skill | Wielded by | When |
|---|---|---|
| `/sofi-boot` | every `qa-*` agent, every session | First move, always — orients on `STATE.md`+`HANDOFFS.md`+`CONTEXT.md` before touching a merged build or a frozen prototype. No Quality agent tests, audits, or quarantines on memory of a prior project's build. |
| `/sofi-delegate` | `qa-lead` | Turning the incoming Gate-5 ticket (from `bck-lead`, once all four Gate-4 rooms are merged) into a paste-ready four-field Work Order for `qa-test-architect` first, always — nothing else in the room dispatches before the strategy exists. |
| `/sofi-team` | `qa-lead` | Confirming which of the room's six specialists owns a given finding before drafting the next Work Order — especially the judgment call on whether a Tier-A pass^k leg belongs with `qa-automation-engineer` (script-drivable) or `qa-manual-explorer` (needs real human variation across runs). |
| `/sofi-gate` | `qa-lead` | The real Gate-5 exit decision — this room owns the gate outright. Wraps `sofi gate-check --gate 5` (mechanical) and `gtw-gatekeeper`'s fresh-context adversarial verdict (never self-graded) from `playbooks/gate-5-quality-procedure.md` steps 9-10. `qa-lead` also runs `sofi gate-check --gate 5 --room 09-security` mid-procedure to confirm the squad partner's contribution has landed before she assembles the aggregate. |
| `/sofi-handoff` | every `qa-*` agent | Closing ritual on every artifact: checkpoint → `CONTEXT.md` → `STATE.md` `head_sha` → next ticket in `HANDOFFS.md`. Runs after every specialist's report is accepted by `qa-lead`, and again at the room's own Gate-5 close (step 12 of the core playbook). |
| `/sofi-audit` | every `qa-*` specialist (self-check before handing to `qa-lead`) | A mechanical layer sweep (`security`/`wiring`/`design` modes as relevant) run before a specialist calls a draft done — `qa-automation-engineer` runs it to catch an obvious untested wiring seam before `qa-lead`'s gate-check even starts; `qa-design-auditor` runs it (via `uiux_pipeline.py gate`) as her literal first step. |
| `/sofi-fix` | `qa-lead` (routes, doesn't execute alone) | Routing a Critical/High finding — a coverage gap, a design deviation, a perf breach — to the owning Build room's Lead. `qa-lead` never fixes a finding herself; the owning specialist applies the fix, checkpoints, and hands back for re-test. |
| `/sofi-report` | `qa-lead` | Writing the Gate-5 accountability check-in for `brd-cqo`/`brd-ceo` as a durable, evidence-backed record — coverage figures, perf verdict, pass^k results, design audit summary, `09-security`'s folded-in findings — rather than letting the verdict live only in a chat message. |
| `/sofi-secure` | none directly (routed) | Any finding this room surfaces that looks security-shaped (an authz gap `qa-manual-explorer` stumbles into mid-exploration, a data-exposure pattern `qa-automation-engineer`'s contract tests reveal) routes immediately to `09-security` via `qa-lead` → `sec-lead` — `10-quality` flags it, `09-security` owns the formal response. |
| `/sofi-spec-review` | none directly (consumer, not owner) | `04-architecture`'s `arc-review-architect` owns this skill; `10-quality` is a frequent *subject* of it when a feature's test-coverage or quality-gate surface needs a cross-layer verdict — `qa-lead` receives the request through her own gateway, never triggers the review itself. |
| `/sofi-reflect` | none directly | The room contributes heavy signal (a recurring flake-shaped pattern, a repeated design-drift category, a Tier-A surface that keeps landing borderline pass^k results) but does not execute the distillation itself — that's `knw-reflector`'s job on `brd-ceo`'s schedule. This room's own `HANDOFFS.md` history, especially its BLOCK verdicts and their named gaps, is exactly the signal that feeds it. |
| `/sofi-feature` | none directly | The end-to-end feature loop calls into this room's Gate-5 procedure as one stage among several; `qa-lead` participates when addressed, but does not initiate a whole-feature pass herself — that stays the caller's (typically `brd-ceo` or `arc-review-architect`) decision. |

## Rules

- `10-quality` is the one room whose `/sofi-gate` invocation is a real, unscoped owner-room exit decision — `qa-lead` runs the full aggregate check, not a `--room` scoped contribution, because Gate 5 belongs to this room outright (`company/nexus/gates.yaml`, `owner_room: 10-quality`).
- `qa-lead` never runs `/sofi-secure` as the owning decision — a security-shaped finding is flagged and routed, and `09-security`'s own escalation path (the security spur) takes it from there, bypassing this room's chain entirely.
- Every skill invocation still obeys the Oracle Loop (Teaching VII) at its own decision points — a borderline pass^k result on a Tier-A surface, or a disputed design deviation the building room insists is intentional, is exactly the kind of decision point `gtw-external-reviewer`'s desk exists for, reached through `qa-lead`, never by a specialist mid-draft.
- `/sofi-delegate` always dispatches `qa-test-architect` first in a fresh Gate-5 pass — every other specialist's work depends on her strategy existing; dispatching them in parallel with her, rather than behind her, is a sequencing defect this room's own playbook exists to prevent.
