# Room 12-observability — Skills

> `12-observability` owns Gate 8 outright — the smallest room-to-gate ratio in the company — and it's the one gate that doesn't really close once and stop: the standing reporting cadence in `playbooks/gate-8-observe-procedure.md` keeps running for as long as the project stays in production, and the incident procedure in `playbooks/incident-response-postmortem.md` can fire at any moment, independent of that cadence. This is the room's own reading of when each `/sofi-*` skill fires, not a duplicate of the skill files themselves (`.claude/skills/sofi-*/`).

## Skills this room wields, and when

| Skill | Wielded by | When |
|---|---|---|
| `/sofi-boot` | every `obs-*` agent, every session | First move, always — orients on `STATE.md`+`HANDOFFS.md`+`CONTEXT.md` before touching a live production system or a prior incident's history. No Observability agent triages, reports, or files a re-open on memory of a prior cycle's numbers. |
| `/sofi-delegate` | `obs-lead` | Turning the incoming Gate-8 ticket (from `ops-lead`, once Gate 7 tags) into a paste-ready four-field Work Order for `obs-sre` first, always — instrumentation and alerting both depend on the SLI/SLO set existing before either one starts. |
| `/sofi-team` | `obs-lead` | Confirming which of the room's five specialists owns a given signal before drafting the next Work Order — especially the judgment call on whether a live production condition is still a monitoring-and-instrumentation question (`obs-monitoring-engineer`) or has crossed into a declared incident (`obs-incident-commander`). |
| `/sofi-gate` | `obs-lead` | The real Gate-8 exit decision — this room owns the gate outright. Wraps `sofi gate-check --gate 8` (mechanical) and `gtw-gatekeeper`'s fresh-context adversarial verdict (never self-graded) from `playbooks/gate-8-observe-procedure.md` steps 9-10. For a money/auth/PII surface's breach report specifically, `obs-lead` requests the family-diverse oracle desk read (Article 03 judge-bias guard) rather than a single-model verdict. |
| `/sofi-handoff` | every `obs-*` agent | Closing ritual on every artifact and every incident: checkpoint → `CONTEXT.md` → `STATE.md` `head_sha` → next ticket in `HANDOFFS.md`. Runs after every specialist's report is accepted by `obs-lead`, after every incident's post-mortem is filed, and again at the room's own standing-cadence report close. |
| `/sofi-audit` | every `obs-*` specialist (self-check before handing to `obs-lead`) | A mechanical layer sweep run before a specialist calls a draft done — `obs-monitoring-engineer` runs it (`search`/`wiring` modes via `sofi_scan.py`) to locate an existing instrumentation hook before writing a redundant one; `obs-alerting-engineer` runs it to confirm no orphaned alert rule already exists with a stale or missing runbook. |
| `/sofi-fix` | `obs-lead` (routes, doesn't execute alone) | Routing a Critical/High finding — an SLO consistently missed, an alert with no runbook discovered mid-audit — to the owning specialist inside this room, or to another room's Lead when the root cause traces outside Observability (a schema issue → `dat-lead`, an infra ceiling → `arc-lead`). `obs-lead` never fixes a finding herself. |
| `/sofi-report` | `obs-lead` | Writing the Gate-8 accountability check-in for `brd-ceo` as a durable, evidence-backed record — SLO figures, error-budget status, drop-off findings, incident summaries when applicable — rather than letting the state of production live only in a chat message. |
| `/sofi-secure` | none directly (routed) | Any incident this room's triage recognizes as security-shaped routes immediately to `09-security` via `obs-lead` → `sec-lead` per `playbooks/incident-response-postmortem.md` step 4a — `12-observability` flags and hands off, `09-security` owns the formal response from that point. |
| `/sofi-spec-review` | none directly (consumer, not owner) | `04-architecture`'s `arc-review-architect` owns this skill; `12-observability` is a rare *subject* of it when a production incident reveals a cross-layer design flaw worth a formal 4-pillar read — `obs-lead` receives the request through her own gateway, never triggers the review herself. |
| `/sofi-reflect` | none directly | The room contributes heavy signal (a recurring incident failure-mode, a repeated false-positive alert pattern, a journey stage that keeps drifting) but does not execute the distillation itself — that's `knw-reflector`'s job on `brd-ceo`'s schedule. This room's `DECISIONS.md` post-mortem entries and `HANDOFFS.md` Gate-1 re-open history are exactly the signal that feeds it. |
| `/sofi-feature` | none directly | The end-to-end feature loop can call into this room's Gate-8 procedure as its final stage; `obs-lead` participates when addressed but does not initiate a whole-feature pass herself — that stays the caller's decision. |

## Rules

- `12-observability` is the one room whose `/sofi-gate` invocation runs on a *standing cadence*, not a single pass-through — `obs-lead` re-opens the reporting cycle every time the project remains in production, rather than treating one clean `SLO_Report.md` as the gate being permanently closed.
- `obs-incident-commander` never waits for `/sofi-delegate` or any Work Order ritual to declare command on a confirmed live incident — command authority is hers the instant the signal is confirmed real, and the incident procedure (`playbooks/incident-response-postmortem.md`) runs independent of and faster than the room's normal skill-invocation rhythm.
- `obs-lead` never runs `/sofi-secure` as the owning decision — a security-shaped incident is flagged and handed off immediately, and `09-security`'s own escalation path (the security spur) takes it from there, bypassing this room's chain entirely.
- Every skill invocation still obeys the Oracle Loop (Teaching VII) at its own decision points — a disputed SLO-breach root cause, a borderline drop-off significance call on a money/auth/PII journey stage — is exactly the kind of decision point `gtw-external-reviewer`'s desk exists for, reached through `obs-lead`, never by a specialist mid-draft.
- `/sofi-delegate` always dispatches `obs-sre` first in a fresh Gate-8 pass — both `obs-monitoring-engineer`'s instrumentation and `obs-alerting-engineer`'s alert rules depend on her SLI/SLO set existing first; dispatching either of them ahead of her is a sequencing defect this room's own playbook exists to prevent.
