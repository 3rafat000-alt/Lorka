# Room 07-mobile ‚Äî Skills

> Mobile sits alongside `05-backend` and `06-frontend` as one of the three squad rooms building against the same frozen Gate-3 bundle, but it carries no dedicated in-room code reviewer ‚Äî so it leans on `/sofi-audit`+`/sofi-fix` even harder than a room its size normally would, and its `/sofi-gate` usage is a *readiness signal* to the owner room (`bck-lead`), never a self-claimed aggregate exit. This is the room's own reading of when each `/sofi-*` skill fires, not a duplicate of the skill files themselves (`.claude/skills/sofi-*/`).

## Skills this room wields, and when

| Skill | Wielded by | When |
|---|---|---|
| `/sofi-boot` | every `mob-*` agent, every session | First move, always ‚Äî orients on `STATE.md`+`HANDOFFS.md`+`CONTEXT.md` before touching a layer, a Bloc, or a network client. No Mobile agent codes on memory of a prior session's contract. |
| `/sofi-delegate` | `mob-lead` | Turning `arc-lead`'s frozen Gate-3 bundle, or a loop-back ticket from `qa-manual-explorer`/`arc-review-architect`, into a paste-ready four-field Work Order for the specialist who owns that piece ‚Äî the layer skeleton first, always, before state or platform work has anything stable to build against. |
| `/sofi-team` | `mob-lead` | Confirming which of the room's five specialists (or, rarely, which agent in another room) owns a given piece of build work before drafting the Work Order ‚Äî especially when a request straddles two specialties (is a slow list screen a `mob-perf-profiler` jank question or a `mob-state-engineer` rebuild-scope question?). |
| `/sofi-audit` (mode `search`/`wiring`/`security`) | every `mob-*` specialist (self-check before handoff), `mob-lead` (pre-review pass) | A quick static sweep before or during a diff's own review cycle ‚Äî mostly `sofi_scan.py wiring`/`security`/`search` plus `flutter analyze`, the mechanical half of step 5 in `playbooks/gate-4-build-procedure.md`. Locate first, judge second ‚Äî this room leans on it more heavily than a room with a dedicated reviewer, since it's part of what stands in for that missing role. |
| `/sofi-fix` | any `mob-*` specialist, on a finding routed back from `mob-lead`'s review or `/sofi-audit` | Applying a routed, ranked finding ‚Äî never a blind "clean up the file" pass. Pairs directly with a `mob-lead` BLOCK verdict or a `gtw-gatekeeper` escalation result: the finding names the fix, the owning specialist applies it, the diff goes back through review. |
| `/sofi-gate` | `mob-lead` | Not a self-claimed aggregate exit ‚Äî this room's `/sofi-gate` usage is the readiness *signal* sent to `bck-lead`, the named owner room, carrying evidence that this room's own worktree is merged and reviewed. `bck-lead` runs the actual aggregate `sofi gate-check --gate 4` across all squad rooms. |
| `/sofi-handoff` | every `mob-*` agent | Closing ritual on every artifact: checkpoint ‚Üí `CONTEXT.md` ‚Üí `STATE.md` `head_sha` ‚Üí next ticket in `HANDOFFS.md`. Runs after every specialist's draft clears `mob-lead`'s review, and again at the room's own Gate-4 merge. |
| `/sofi-report` | `mob-lead` | Writing the room's Gate-4 contribution as a durable, evidence-backed record in the brain ‚Äî especially useful when the build involved a contested state-management boundary or a platform-channel design that needed the oracle desk's second opinion. |
| `/sofi-spec-review` | `mob-lead` (requests it, via `arc-review-architect`, never runs it directly) | A feature built far enough along in this room to inspect end-to-end and worth a full 4-pillar cross-layer verdict ‚Äî the binding spec-review protocol's "Mobile/Client" pillar is exactly this room's slice of that review. This room is a *consumer* of the skill, not its owner (that's `04-architecture`). |
| `/sofi-secure` | `mob-lead` (requests it, via `sec-lead`) | A threat surface this room's build touches ‚Äî a platform-channel bridge to native code, a secure-storage credential path, an auth-token refresh flow ‚Äî needs a dedicated security pass ahead of Gate 5's formal pentest. Routed cross-room through the Room Isolation Law, never run by a `mob-*` specialist directly. |
| `/sofi-reflect` | none directly | The room contributes signal (a recurring silently-swallowed exception, a repeated leaked-`StreamSubscription` pattern) but does not execute the distillation itself ‚Äî that's `knw-reflector`'s job on `brd-ceo`'s schedule. Worth knowing the room's own `HANDOFFS.md` history, especially any BLOCK verdicts from `mob-lead`'s review pass, is exactly what feeds it. |
| `/sofi-design-taste` | `mob-lead` (consumer only, via `dsn-lead`) | This room builds screens per `Prototype_Spec.md`, not against the taste dials directly ‚Äî taste-dial application to visual polish is `03-design`'s and `06-frontend`'s domain. Relevant here only insofar as a screen's structure has to actually match what the prototype specified, checked, not assumed. |

## Rules

- `07-mobile` never *owns* `/sofi-spec-review` or `/sofi-secure` ‚Äî those are standing cross-gate skills owned by `04-architecture` (`arc-review-architect`) and `09-security` (`sec-lead`) respectively; this room requests them through its Lead like any other room, it never runs the hard-gate verdict itself.
- `/sofi-fix` inside this room is always routed from a ranked finding (a `mob-lead` review BLOCK, a `gtw-gatekeeper` escalation result, a `/sofi-audit` sweep) ‚Äî never a speculative "improve this file" pass with no named defect behind it.
- `/sofi-feature` is a Boardroom-commissioned, cross-gate skill; `07-mobile` supplies the Gate-4 mobile slice of it when `brd-ceo` runs the full loop on a feature, but never triggers the whole arc itself.
- `mob-lead`'s in-room review is not a substitute for `gtw-gatekeeper`'s escalation path on contested diffs, and neither is a substitute for `gtw-gatekeeper`'s Gate-4 adversarial check at the aggregate level where a cross-room dispute needs it ‚Äî the room names explicitly which of the three applies, it never blurs them.
- Every skill invocation still obeys the Oracle Loop (Teaching VII) at its own decision points ‚Äî a genuinely contested state-management boundary or a platform-channel design that survived one round of intra-room mediation is exactly the kind of decision point `gtw-external-reviewer`'s desk exists for, reached through `mob-lead`, never by a specialist directly.
