# Room 05-backend — Skills

> Backend sits at the hinge between "the contract is frozen" (Architecture) and "the build is provably correct" (Quality) — it wields the boot-and-close ritual skills constantly, owns the Gate-4 exit as the named owner room, and leans on `/sofi-audit`+`/sofi-fix` as the mechanical sweep-then-repair loop every specialist runs before a diff is even eligible for `bck-code-reviewer`. This is the room's own reading of when each `/sofi-*` skill fires, not a duplicate of the skill files themselves (`.claude/skills/sofi-*/`).

## Skills this room wields, and when

| Skill | Wielded by | When |
|---|---|---|
| `/sofi-boot` | every `bck-*` agent, every session | First move, always — orients on `STATE.md`+`HANDOFFS.md`+`CONTEXT.md` before touching an endpoint, a service, or a view. No Backend agent codes on memory of a prior session's contract. |
| `/sofi-delegate` | `bck-lead` | Turning `arc-lead`'s frozen Gate-3 bundle, or a loop-back ticket from `qa-automation-engineer`/`arc-review-architect`, into a paste-ready four-field Work Order for the specialist who owns that piece — the domain engineer's service interface first, always, before the API engineer wires the endpoint that consumes it. |
| `/sofi-team` | `bck-lead` | Confirming which of the room's seven specialists (or, rarely, which agent in another room) owns a given piece of build work before drafting the Work Order — especially when a request straddles two specialties (is a webhook-triggered notification a `bck-integration-engineer` shape question or a `bck-queue-engineer` delivery question?). |
| `/sofi-audit` (mode `api`/`security`/`integration`) | every `bck-*` specialist (self-check before handoff), `bck-lead` (pre-review pass) | A quick static sweep before or during a diff's own review cycle — mostly `sofi_scan.py wiring`/`security` and `sofi_verify.py`, the mechanical half of step 5 in `playbooks/gate-4-build-procedure.md`. Locate first, judge second — never opened as a substitute for `bck-code-reviewer`'s actual review. |
| `/sofi-fix` | any `bck-*` specialist, on a finding routed back from `bck-code-reviewer` or `/sofi-audit` | Applying a routed, ranked finding — never a blind "clean up the file" pass. Pairs directly with a `bck-code-reviewer` BLOCK verdict: the SEV report names the fix, the owning specialist applies it, the diff goes back through review. |
| `/sofi-gate` | `bck-lead` | The Gate-4 exit decision — mechanical `sofi gate-check --gate 4` (confirming all four Build-gate rooms, not just this one) plus `gtw-gatekeeper`'s fresh-context adversarial verdict where a cross-room dispute needs it — never a self-graded sign-off. This is the room's single highest-stakes skill invocation, because it's the owner-room report the whole Build gate's exit rests on. |
| `/sofi-handoff` | every `bck-*` agent | Closing ritual on every artifact: checkpoint → `CONTEXT.md` → `STATE.md` `head_sha` → next ticket in `HANDOFFS.md`. Runs after every specialist's draft clears `bck-code-reviewer`, and again at the room's own Gate-4 merge. |
| `/sofi-report` | `bck-lead` | Writing the Gate-4 accountability check-in for `brd-cto` as a durable, evidence-backed record in the brain — especially useful when the build involved a contested money-math edge case or an idempotency design that needed the oracle desk's second opinion. |
| `/sofi-spec-review` | `bck-lead` (requests it, via `arc-review-architect`, never runs it directly) | A feature built far enough along in this room to inspect end-to-end and worth a full 4-pillar cross-layer verdict — most often invoked just before or during Gate 4/5 when `bck-code-reviewer`'s in-room pass surfaces something that looks like it spans layers (a money-math bug that also implicates the schema, say). This room is a *consumer* of the skill, not its owner (that's `04-architecture`). |
| `/sofi-secure` | `bck-lead` (requests it, via `sec-lead`) | A threat surface this room's build touches — an auth flow, a webhook signature scheme, an admin endpoint — needs a dedicated security pass ahead of Gate 5's formal pentest. Routed cross-room through the Room Isolation Law, never run by a `bck-*` specialist directly. |
| `/sofi-reflect` | none directly | The room contributes signal (a recurring idempotency miss, a repeated 302-instead-of-422 regression) but does not execute the distillation itself — that's `knw-reflector`'s job on `brd-ceo`'s schedule. Worth knowing the room's own `HANDOFFS.md` history, especially `bck-code-reviewer`'s BLOCK verdicts, is exactly what feeds it. |

## Rules

- `05-backend` never *owns* `/sofi-spec-review` or `/sofi-secure` — those are standing cross-gate skills owned by `04-architecture` (`arc-review-architect`) and `09-security` (`sec-lead`) respectively; this room requests them through its Lead like any other room, it never runs the hard-gate verdict itself.
- `/sofi-fix` inside this room is always routed from a ranked finding (a `bck-code-reviewer` BLOCK, a `/sofi-audit` sweep) — never a speculative "improve this file" pass with no named defect behind it.
- `/sofi-feature` is a Boardroom-commissioned, cross-gate skill; `05-backend` supplies the Gate-4 backend slice of it when `brd-ceo` runs the full loop on a feature, but never triggers the whole arc itself.
- `bck-code-reviewer`'s in-room review is not a substitute for, and is not substituted by, `gtw-gatekeeper`'s Gate-4 adversarial check — the two operate at different scopes (one diff vs. the whole gate exit) and both are mandatory where they apply.
- Every skill invocation still obeys the Oracle Loop (Teaching VII) at its own decision points — a genuinely contested money-math edge case or an idempotency-key design that survived one round of intra-room mediation is exactly the kind of decision point `gtw-external-reviewer`'s desk exists for, reached through `bck-lead`, never by a specialist directly.
