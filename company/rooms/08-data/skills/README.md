# Room 08-data — Skills

> Data sits behind two other rooms' owner-gates rather than owning one itself — it wields the boot-and-close ritual skills at both legs of its two-posture pass (Gate-3 squad partner, Gate-4 support room), and reaches for the mechanical sweep/repair loop before handing anything to `dat-lead`. This is the room's own reading of when each `/sofi-*` skill fires, not a duplicate of the skill files themselves (`.claude/skills/sofi-*/`).

## Skills this room wields, and when

| Skill | Wielded by | When |
|---|---|---|
| `/sofi-boot` | every `dat-*` agent, every session | First move, always — orients on `STATE.md`+`HANDOFFS.md`+`CONTEXT.md` before touching a frozen schema design or a built backend service. No Data agent writes a migration, a cache design, or a `PII_Map.md` on memory. |
| `/sofi-delegate` | `dat-lead` | Turning the incoming ticket (a Gate-3 squad dispatch from `arc-lead`, or a Gate-4 support ticket from `bck-lead`) into a paste-ready four-field Work Order for the specialist who owns the next piece — `dat-db-engineer` first at both gates, since everything else in the room either validates against or builds on top of the physical schema. |
| `/sofi-team` | `dat-lead` | Confirming which of the room's six specialists owns a given piece of work before drafting the Work Order — especially the judgment call on whether a project's scope actually calls for `dat-analytics-engineer` or `dat-ml-engineer` (neither is dispatched by default; Teaching IV, token economy). |
| `/sofi-gate` | `dat-lead` | This room's contribution check at each gate it touches: mechanical `sofi gate-check --gate 3 --room 08-data` / `--gate 4 --room 08-data` plus `gtw-gatekeeper`'s fresh-context adversarial verdict on this room's slice — never a self-graded sign-off. Wraps steps 5/12-13 of `playbooks/gate-3-4-data-layer.md`. `dat-lead` signs the room's *contribution*; he does not run the aggregate Gate-3/Gate-4 exit decision itself — that stays `arc-lead`'s and `bck-lead`'s job as the named owner-room Leads. |
| `/sofi-handoff` | every `dat-*` agent | Closing ritual on every artifact: checkpoint → `CONTEXT.md` → `STATE.md` `head_sha` → next ticket in `HANDOFFS.md`. Runs after every specialist's draft is accepted by `dat-lead`, and again at each leg's own close. |
| `/sofi-audit` | every `dat-*` specialist (self-check before handing to `dat-lead`) | A mechanical layer sweep (`db`/`api`/`wiring` modes as relevant) run before a specialist calls a draft done — `dat-db-engineer` runs it to catch an obvious missing index or an unindexed foreign key before `dat-lead`'s gate-check even starts; `dat-privacy-officer` runs it to catch an obviously-missed personal-data field pattern before the manual field-by-field walk. |
| `/sofi-fix` | `dat-lead` (routes, doesn't execute alone) | Routing a `/sofi-audit` or `/sofi-spec-review` finding touching this room's surface to the owning specialist — `dat-lead` never lets a finding sit unrouted, and never fixes it himself; the owning specialist applies the fix, checkpoints, and hands back. |
| `/sofi-report` | `dat-lead` | Writing the room's Gate-3/Gate-4 accountability check-in for `brd-ceo`/`brd-cto` as a durable, evidence-backed record — the migration-validation feedback, the `PII_Map.md` summary, the executed-migration + cache + sync contribution — rather than letting it live only in chat. |
| `/sofi-secure` | none directly (routed) | PII/encryption findings that carry a security dimension route to `09-security` via `dat-lead` → `sec-lead` (the security spur, immediate, no queue) — `08-data` classifies and maps, `09-security` (through `sec-lead`) owns the formal compliance response and any veto. |
| `/sofi-spec-review` | none directly (consumer, not owner) | `04-architecture`'s `arc-review-architect` owns this skill; `08-data` is a frequent *subject* of it when a feature's data-layer surface needs a cross-layer verdict — `dat-lead` receives the request through his own gateway (any room's Lead may address `arc-review-architect`), never triggers the review itself. |
| `/sofi-reflect` | none directly | The room contributes signal (a Gate-3 PII gap, a recurring cache-stampede-shaped finding, a migration rollback rejection) but does not execute the distillation itself — that's `knw-reflector`'s job on `brd-ceo`'s schedule. Worth knowing the room's own `HANDOFFS.md` history is exactly what feeds it. |

## Rules

- `08-data` never invokes `/sofi-feature` or `/sofi-gate` as the *owning* decision for a whole gate — it owns no gate outright (`gates.yaml`: Gate 3 owner is `04-architecture`, Gate 4 owner is `05-backend`); this room's `/sofi-gate` use is always scoped to `--room 08-data`, its own contribution, never the aggregate exit.
- `dat-analytics-engineer` and `dat-ml-engineer` are never dispatched by default through `/sofi-delegate` — `dat-lead` confirms the ticket names their scope first (`/sofi-team`'s judgment call), or the dispatch is a token-economy defect.
- Every skill invocation still obeys the Oracle Loop (Teaching VII) at its own decision points — a contested retention-window call, an eval-suite result that's borderline against its baseline, or a cache-invalidation design for a money/PII-adjacent key is exactly the kind of decision point `gtw-external-reviewer`'s desk exists for, reached through `dat-lead`, never by a specialist mid-draft.
