# SOFI Development Plan — Synthesis of the AI Guides Corpus

## Context

This plan synthesizes all 40 per-article analyses in `.claude/docs/ai-guides/` — write-ups of pieces from aiwithmo.com (mostly its `/prompts` section, one `/blog` post) covering AI agent orchestration, memory/context systems, prompt engineering, model releases, and a number of unrelated consumer-product announcements — into concrete recommendations for developing the SOFI agent team (`engine/`, `.claude/agents/`, `engine/protocols/`). This supersedes an earlier partial pass that only covered 22 of the 40 files; every article now has a per-file analysis on disk and every one was read for this synthesis (see Coverage, below). This is a planning document for the user to review before deciding what to implement — not a sales pitch. Every claim below is cited to a specific article filename, and disagreements/tradeoffs are called out rather than smoothed over.

## Already validated

Patterns the articles independently (the aiwithmo.com author had no visibility into SOFI's design) confirm SOFI already has right:

- **RCCF delegation as a sound structured-prompting format.** The Role/Context/Task/Format structure in `improve-asking.md` and the goal-based ("deliver a prioritized action plan," not "read file A, then B") delegation framing in `autonomous-agents-cowork-code.md` both independently converge on what `engine/protocols/01-delegation-rccf.md` already enforces via its Command field's in-bounds/out-of-bounds/success-metric structure. `perfect-prompt-engineer.md`'s "meta-prompt persona that only drafts, never executes" independently matches `/sofi-delegate`'s purpose.
- **The git-backed, multi-file project brain.** `ai-os-context-engineering-local-files.md`'s three-layer file structure (instructions / knowledge / working-state), `cowork-memory-system.md`'s single-`MEMORY.md`-plus-short-sessions loop, `context-engineering-vibe-coding.md`'s four-file discipline (ROADMAP/ARCHITECTURE/CONTEXT/PROGRESS), and `five-levels-ai-second-brain.md`'s Level 1–2 "router file + evergreen-vs-changing separation" all describe the same shape SOFI already implements as `STATE.md`/`CONTEXT.md`/`DECISIONS.md`/`HANDOFFS.md` — split across four files rather than one, arguably more disciplined against merge contention (per `cowork-memory-system.md`'s own reasoning about why single files become bottlenecks).
- **Gate-sequenced, phase-based execution with a verification bar before advancing.** `gsd-framework.md`'s Plan/Execute/Review phases with context resets and "clean exit code + git commit before done," and `autonomous.md`'s `/gsd-do`/`/gsd-autonomous` phase runner, both match SOFI's 0→8 gate lifecycle with `gates.py` artifact validation — SOFI's version is described in both articles' own relevance sections as the more rigorous, machine-checked implementation of the same idea.
- **Specialized, isolated-context workers over one broad context window.** `cowork-subagent-coordination.md`'s "isolated context per worker" and `from-sequential-to-parallel-ai-agents.md`'s Planner/Dispatcher/Worker/Synthesizer graph both map directly onto CEO (planner) → Advisor (dispatcher) → Specialist (worker) → Advisor (synthesizer), and onto SOFI's per-tier worktree isolation.
- **The economic model-routing ladder, decoupled from an effort/thinking dial.** `advisor-mode.md`'s "expensive model for planning, cheap model for execution," `claude-code-glm.md`'s Architect/Executor split, and `ai-model-selection-2026.md`'s explicit "decouple tier from thinking mode" framing all describe, at the single-session level, what `engine/routing/routing.yaml`'s haiku→sonnet→fable→opus ladder plus its separate effort dial already does at the org level, per-role, all the time.
- **Agent-authored orchestration scripts with adversarial verification for large, unclear-decomposition work.** `ultracode-agent-orchestration.md`'s "Claude writes the coordination script, a separate cluster tries to refute the first cluster's findings" is, per that article's own relevance note, functionally identical to SOFI's `Workflow` tool pattern — independent convergent validation rather than a new idea to adopt.
- **Plan-approval-before-execution as governance.** `claude-agent-teams.md`'s "teammates submit plans to the lead for approval before executing" and `higgsfield-supercomputer.md`'s "show me the plan before executing" both match SOFI's gate-check + Definition-of-Done pattern.
- **Project-level context isolation, no cross-bleed.** `projects-feature.md`'s "one workspace per client/domain, never mix" is exactly SOFI's `PRJ-XXXX` isolation rule.

## Genuine gaps found

Ranked most-actionable first (low effort / high value first; long-tail low-priority items last).

1. **No codified grounding / anti-hallucination doctrine in `engine/protocols/00-operating-system.md`.** Surfaced by `grounding-rules-instructions-file.md`, which lays out six concrete rules (source-of-truth, honesty about not knowing, no fabrication, mandatory citation, separating stated knowledge from inference, surfacing conflicting sources instead of silently picking one) that agent-mode tools are expected to follow before answering — reinforced by `notebooklm.md`'s "answer strictly from the cited documents, flag what you can't source" pattern. SOFI expects `file:line` citation in review/audit contexts by convention, but there is no universal, formal rule binding every agent's every claim. **Next step**: add a "Grounding" section to `00-operating-system.md` codifying these six rules as part of the universal contract, and require RCCF Context blocks / the external review-desk payload to instruct the receiving agent to answer strictly from cited brain files and frozen artifacts, flagging anything it cannot source. This was flagged as the single most concrete, low-effort, high-value gap across the corpus.

2. **No scheduled self-improvement / "dreaming" loop reviewing `HANDOFFS.md` history to update doctrine.** Surfaced by `claude-agent-teams.md`, which describes a periodic process (added to Anthropic's own Cowork product May 7, 2026) where agents review past sessions, extract behavioral patterns, and update memory automatically — with a measurable, cited gap between freshly-configured and month-old agents. SOFI's OODA loop has a conceptual "Reflection" layer but no actual scheduled job reading `HANDOFFS.md`/`DECISIONS.md` history across projects to propose doctrine updates. **Next step**: design a periodic (per-gate-close or nightly) job that reads recent `HANDOFFS.md`/`DECISIONS.md` entries, extracts recurring friction points or successful patterns, and proposes — not auto-applies — updates to `engine/protocols/` or a role's spec for human review, consistent with SOFI's existing rule that the CEO doesn't write code (doctrine changes need the same discipline).

3. **`/sofi-secure` has no adversarial self-verification pass, and findings lack a ready-to-delegate execution plan.** Surfaced by `claude-security-beta.md` (Claude Security's find→adversarially-self-challenge→patch flow, cutting false positives before a report ships) and `security-scan.md` (Code Guardian's dynamic scan-phase-count-by-project-size, plus an "Execution Plan" fix checklist attached to every finding, pre-formatted for a downstream coding agent). SOFI's `/sofi-secure` and `/sofi-audit` are already grep-first and severity-ranked but don't formalize either of these two steps. **Next step**: add a self-challenge pass to the Security & Compliance Architect's flow (re-verify each finding before ranking severity+confidence), and extend each SEV finding with a concrete, specialist-scoped Execution Plan block that `/sofi-fix` can delegate directly, closing the gap between "found" and "assigned-fixable" in one step.

4. **`/sofi-delegate` doesn't branch to clarifying questions on underspecified tickets, and doesn't enforce a single frozen brief.** Surfaced by `perfect-prompt-engineer.md` (a "prompt engineer" persona asks clarifying questions before drafting if the raw ask is too vague, otherwise drafts) and `lovable-dev-guide.md` (paste one complete PRD as the first message, never drip-feed instructions afterward). SOFI's RCCF already implies completeness but doesn't formally gate on it. **Next step**: harden `/sofi-delegate` to explicitly branch — if Role/Context/Command/Format can't be filled with real specifics, emit clarifying questions instead of a half-formed block; once complete, treat the RCCF block as frozen (no incremental instruction drip after spawn).

5. **No "extract-to-template" step to grow the delegation library from lived corrections.** Surfaced by `reverse-prompt-engineering.md`: once a live back-and-forth with a specialist converges (5+ correction rounds), run an extraction pass distilling the corrected exchange into a reusable master prompt, rather than only hand-authoring specs cold. **Next step**: add an extract-to-template step to `/sofi-delegate` — after any RCCF spawn needing 2+ correction rounds, fold the corrections into an updated reusable template for that agent+task-type, stored under `engine/protocols/` or the agent's spec, so the org's delegation library compounds from real work.

6. **Brain files carry no structured metadata for mechanical querying.** Surfaced by `obsidian-claude-second-brain.md`'s YAML-frontmatter-on-every-note pattern, which turns a folder of plain Markdown into a queryable database (e.g., "all tickets with status=blocked"). SOFI's `STATE.md`/`HANDOFFS.md` are grep-only today. **Next step**: add optional lightweight frontmatter (`status`, `gate`, `type`, `date`) to ticket/entry blocks so `sofi` tooling can query structurally instead of only via grep — low effort, directly reuses a technique the article demonstrates working at scale.

7. **Per-role Operating Prompts have no explicit "boundaries/constraints" subsection.** Surfaced by `projects-feature.md`'s system-instruction template, which treats "what this must NOT do" as a first-class field alongside role/context/rules — SOFI currently leaves this implicit in gate-bar/escalate sections. **Next step**: add a boundaries/constraints checklist item to `engine/agents/<role>/` Operating Prompts (what this agent must not do, must not pull from other projects) when next revised.

8. **No structured "Motion Spec" sub-template for UI animation delegations.** Surfaced by `ai-ui-animation-engineering.md`'s four-part prompt structure (persona → library choice → easing/duration/stagger/trigger → performance rules: transform/opacity only, 60fps, `prefers-reduced-motion`) — a ready-made checklist for Gate 4's TTI<2s bar and accessibility requirements. **Next step**: add this sub-template to the `sofi-design-taste` skill or the RCCF delegation template so any agent briefed on animation work gets the same structured constraints instead of a vague "add animation" instruction.

9. **No formalized brand-capture interview procedure at Gate 2.** Surfaced by `brand-skill-duide.md`'s 5-step sequential interview (foundational attributes → color palette → typography → layout/aesthetic → compile into a structured skill file) for turning tacit brand knowledge into a durable, reusable artifact. SOFI has `sofi-design-taste`'s dials + brand presets but no explicit intake procedure. **Next step**: have the UI/UX lead or `sofi-design-taste` run this 5-step interview once per project at Gate 2, storing the resulting brand spec (attributes, color table, font map, layout rules) in `_context/` so downstream designers/frontend agents reference one source instead of re-deriving brand rules per screen.

10. **No task-size/effort hint alongside model-tier choice, and routing escalations don't log their trigger-evidence.** Surfaced by `claude-opus-4-8-release.md`'s effort-slider concept (low/medium/high/xhigh/max/ultracode — verify within one model call, not just escalate across models) and `claude-fable-5-release.md`'s transparent-fallback pattern (tell the requester *why* a request got rerouted, not just where). **Next step**: expose a light/standard/deep effort hint in `/sofi-delegate` alongside the existing tier choice, and have routing-escalation logs capture the trigger evidence, not just the destination tier. Treat both articles' specific benchmark numbers and model-version claims as unverified marketing, not a basis for changing `routing.yaml`'s actual model IDs.

11. **Gate 3→4 handoff doesn't formally require the architecture artifact be broken into micro-tasks with explicit verification criteria.** Surfaced by `claude-code-glm.md`: the article's real lesson is that plan *quality* — not model tier alone — is what lets a cheap executor run long unsupervised stretches without drifting. **Next step**: when routing Build-gate (Gate 4) work, ensure the upstream architecture/spec artifact (from Fable/Opus at Gate 3) is explicitly broken into micro-tasks with success criteria before handoff, rather than leaving decomposition implicit.

12. **The Advisor-gateway model trades speed for governance versus the field's direct-peer-communication pattern — a deliberate tradeoff to stay aware of, not necessarily a gap to fix.** Surfaced by `claude-orchestration-parallel.md`, whose "Agent Teams" tier describes parallel agents communicating directly with each other and the lead with no coordinator overhead — the opposite of SOFI's mandatory Advisor-gateway for every cross-tier handoff. This is a deliberate SOFI design choice prioritizing traceability/governance, and should not be "fixed" reflexively. **Next step**: no change now. If the Advisor-gateway is ever measured as a real bottleneck (not just theoretical overhead), revisit whether same-tier direct comms could extend to adjacent-tier fast paths for low-stakes tickets while keeping the gateway mandatory for anything gate-crossing or high-stakes.

13. **No complexity threshold for skipping full RCCF+Advisor-gateway ceremony on trivial single-file tickets.** Surfaced by `cowork-subagent-coordination.md`, whose automatic-complexity-evaluation pattern only parallelizes/formalizes past a size bar — trivial tasks stay lightweight. SOFI's gate/ticket model currently routes every ticket through full CEO/Advisor delegation regardless of size. **Next step**: not urgent — no evidence yet that delegation overhead is a real bottleneck on small tasks; worth a threshold rule only if it's measured as one.

14. **No situation-based routing table mapping task type to the doc to consult first.** Surfaced by `memory-map.md`, which describes a `memory-map.md` lookup table ("adding a UI component → `ui-guidelines.md`," "modifying the database → `schema-rules.md`") that prevents agents from randomly browsing the codebase to figure out what to read. SOFI's per-role specs mostly do this implicitly via their own "Context — read before acting" sections. **Next step**: low priority — a short routing table in `CLAUDE.md` or a new `engine/protocols/situation-routing.md`, only if agents are observed guessing which doc to read.

15. **No documented "$0 MVP fast-path" preset for early-stage client demos.** Surfaced by `ship-your-first-project-for-zero.md`'s zero-cost free-tier stack (GitHub + Vercel + Supabase) for spinning up a throwaway MVP before committing to full staging/prod infrastructure. **Next step**: document this as an optional preset in `engine/protocols/local-domains.md` or a `new-project.sh` flag that DevOps/CEO can offer for early client demos ahead of Gate 6/7 staging.

16. **No plan yet for scaling the brain past Level 1/2 as `CONTEXT.md`/`DECISIONS.md` history grows.** Surfaced by `five-levels-ai-second-brain.md`'s 5-level maturity ladder, with its own explicit warning against over-engineering — don't jump to semantic search or knowledge graphs before the pain is real (the article cites "30+ notes" as its own signal threshold). **Next step**: not urgent for any project yet; when a project's `CONTEXT.md`/`DECISIONS.md` history grows large, introduce a lightweight semantic-search layer before considering full knowledge-graph tooling.

17. **Unconfirmed whether Claude Code's Fast Mode is usable for SOFI's haiku-tier ops, and no defensive cache-health monitoring.** Surfaced by `ai-updates-roundup-may-2026.md` (Fast Mode's ~2.5x speed lever, unverified for subagent spawns) and `claude-codes-limit.md` (a real Anthropic prompt-caching regression that silently forced full token retransmission every turn — a failure mode invisible until limits/cost spike). **Next step**: verify whether `/fast` is usable inside SOFI subagent spawns and document it in `routing.yaml` if so; separately, add a lightweight cache-health check (e.g. in `sofi doctor` or a routing-cost log) that flags abnormal token-consumption growth per session as a leading indicator of upstream caching regressions.

18. **RCCF context-loading could be more explicit about naming specific knowledge files.** Surfaced by `ai-os-context-engineering-local-files.md`'s "load only relevant knowledge files per task" discipline. **Next step**: minor refinement — when delegating via RCCF, name the specific file(s) an agent should load rather than pointing at the whole `engine/agents/**` tree. Refinement of the existing contract, not a structural gap.

19. **No explicit anti-pattern caution against relying on a model still in public testing for critical workflows.** Surfaced by `ai-model-selection-2026.md`'s named anti-pattern ("don't build critical production workflows on a model still in public testing"). **Next step**: add this as a one-line caution in `engine/protocols/00-operating-system.md` or a `routing.yaml` comment, so new agents/roles inherit the same discipline when a new model tier is piloted before promotion.

## Not applicable

7 of the 40 articles are pure product announcements or consumer tools with no transferable agent-orchestration or engineering technique, self-identified as "reference only" in their own Relevance sections: `carousel-os.md` (an Instagram-carousel generation tool), `onemenu-mac-app.md` (a macOS menu-bar utility), `claude-code.md` (a bare links/course-upsell page), `google-io-2026-gemini.md` (a Gemini product-launch roundup), and `openclaw-and-the-alternatives.md` (an OpenClaw-vs-Codex-vs-Cowork product-landscape piece), among others. A further handful — `ai-updates-roundup-may-2026.md`, `claude-opus-4-8-release.md`, `claude-fable-5-release.md`, `claude-security-beta.md` — are primarily release/marketing announcements where the bulk of the content (benchmarks, pricing, funding, version numbers) is not independently verifiable and shouldn't be trusted verbatim for SOFI decisions; each yielded exactly one small transferable fragment (folded into the gaps list above) rather than a technique worth treating as validated doctrine.
