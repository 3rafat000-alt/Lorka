# Evaluating and Verifying LLM Agent Output

## State of the art (2025-2026)

**Eval harnesses have replaced unit tests as the agent QA unit.** Anthropic's own doctrine (`demystifying-evals-for-ai-agents`) separates *task* (input + success criteria), *grader* (scoring logic, can be plural), and *harness* (runs trials, isolates state, aggregates). Three grader types compose: code-based (cheap, brittle), model-based (flexible, needs calibration), human (gold standard, slow). Key metrics are `pass@k` (can it succeed at all) vs `pass^k` (does it succeed *reliably*) — agents are graded on consistency, not just capability. A recurring finding: 0% pass rate across many trials usually means a broken task/grader, not an incapable agent — teams under-invest in auditing their own graders.

**LLM-as-judge is now a studied, biased instrument, not a free lunch.** 2025-2026 literature (arXiv 2604.23178 "Judging the Judges", 2606.19544 "Reliability without Validity") documents systematic failure modes: position bias, verbosity/length bias, self-enhancement bias (judge favors output from its own model family), and authority bias. The fix pattern is consistent across sources: use a *different model family* as judge than generator, calibrate the judge against a human-labeled gold set (Cohen's kappa threshold before trusting it), give the judge an explicit "Unknown/insufficient evidence" escape hatch instead of forcing a verdict, and treat judge-human disagreement as a finding to investigate, not noise to average away.

**Multi-agent debate does not reliably catch errors — it can amplify them.** "The Confident Liar" (arXiv 2606.10296) shows debate can propagate confidently-stated wrong claims because agents weight *rhetorical confidence*, not correctness; recommends combining debate with log-probability signals and an independent judge rather than trusting consensus. "The Deliberative Illusion" (2606.03032) documents *stance homogenization* and *factual attrition* — agents converge for social reasons, losing correct minority positions. Net finding: debate is a *diversity generator*, not a verifier; it only helps if paired with an independent, structurally-adversarial check (opposing roles + a separate judge), e.g. the "courtroom" pattern (PROClaim, arXiv 2603.28488) where roles are fixed (prosecutor/defender/judge) rather than free-form.

**Verifier/process-reward models are the SWE-bench-era answer for code.** Instead of one pass/fail signal at the end (outcome reward model), process reward models (PRMs) score intermediate reasoning/diff steps, catching wrong-but-plausible-looking code earlier and cheaper than running the full test suite. Generative PRMs that reason with explicit chain-of-thought + code execution before scoring now outperform plain LLM-as-judge using far less labeled data (arXiv reports ~1% of process labels). For coding specifically, the dominant pattern is: **don't trust the agent's self-report** ("tests pass", "booking confirmed") — verify against ground truth (run the actual test suite, query the actual DB row, execute the actual diff) — this is Anthropic's explicit guidance in both eval and harness posts.

**Confidence calibration in LLMs is structurally broken and getting worse-understood, not better.** Verbalized confidence ("I'm 90% sure") is empirically miscalibrated — models are overconfident especially on things they encode little information about, and an agent that already committed to an action is structurally biased to justify rather than critique it (self-serving bias, not just noise). 2025-2026 work (DINCO, arXiv 2509.25532) tries to correct this by asking the model to score self-generated distractors and normalizing across them, rather than trusting one verbalized number. Bottom line for practitioners: **never gate an irreversible action on a model's self-reported confidence** — use behavioral/outcome signals (does the artifact exist, does the test pass) instead.

**Long-running agent harnesses (Anthropic, `effective-harnesses-for-long-running-agents`) verify behaviorally, not by trusting the transcript**: browser automation instead of "the code looks right," a structured feature-status file the agent can only flip `passes: true` on via an external check (can't edit/delete tests), one feature per session to bound blast radius, and git as the rollback substrate.

## Concrete techniques worth adopting

1. **Outcome verification over self-report** — never accept "done/booked/tests pass" from the acting agent; check the actual state (git diff exists, test exit code, DB row, file on disk). Why: closes the single biggest plausible-but-wrong gap; this is Anthropic's #1 repeated finding across both eval and harness posts.
2. **Judge-generator family diversity** — when using LLM-as-judge, use a different model/family than the one that produced the artifact, or at minimum a different prompt persona with adversarial framing. Why: self-enhancement bias is empirically the largest, most consistent judge bias.
3. **Judge calibration against a gold set at setup time** — before trusting any model-based grader, score 20-50 known-answer cases and require kappa above a threshold before it grades real work. Why: an uncalibrated judge is worse than no judge — it manufactures false confidence.
4. **"Unknown" escape hatch in every judge rubric** — instruct the judge/verifier to output "insufficient evidence" rather than forcing pass/fail. Why: without it, judges hallucinate justifications for whichever side looks more fluent (verbosity bias).
5. **Adversarial self-verify with a fixed opposing role, not open debate** — pick one agent to argue "this is wrong," another to defend, a third (different model) to decide — not N agents free-associating. Why: free-form multi-agent debate measurably homogenizes toward confident-but-wrong claims; role-fixed adversarial structure resists that.
6. **pass^k over pass@k for anything going to production** — run the same task k times; require *all* k to succeed, not just one. Why: pass@1 or best-of-k hides flaky/non-reproducible correctness, which is exactly what "plausible but wrong" looks like statistically.
7. **Process-level verification for code, not just outcome** — score/check the diff or intermediate steps (does this hunk touch the right files, does it match the frozen contract) before running the full test suite. Why: cheaper, catches drift earlier, and generative PRMs now beat LLM-as-judge on this specific task.
8. **Never gate irreversible actions on verbalized confidence** — route confidence-adjacent decisions (should I escalate, should I ship) through behavioral proxies (did the mechanical check exit 0) instead of the model's self-rated certainty. Why: verbalized confidence is proven miscalibrated and self-serving once the model has committed to an answer.
9. **Read the transcripts, not just the score** — periodically sample full trajectories behind a grader's PASS to catch reward-hacking / grading bugs (Anthropic explicitly flags "0% pass" as usually a broken grader, and "grading bugs accepted only exact numeric matches" as a real incident). Why: a grader nobody audits silently drifts into either false-lenient or false-strict.

## How it applies to SOFI

**What SOFI already does right:** the mechanical-gate pattern (`sofi_verify.py` must exit 0 before "done" is declared) is exactly the outcome-verification discipline the frontier recommends — it doesn't trust an agent's "I finished" claim, it runs lint/view/route/flutter/assets checks and gates on exit code. `thinking-and-work.md`'s "Self-verify... ask what would make this wrong" plus spawning `cavecrew-reviewer` for high/max tasks is a lightweight adversarial-check pattern. `research-and-internet.md`'s citation rule (`claim [source: url, fetched date]`, "I read this" vs "I infer this") is a real anti-hallucination discipline most agent frameworks lack entirely. `/sofi-spec-review`'s two-phase design (cheap grep/haiku sweep → Fable hard gate) mirrors the graduated-cost verifier pattern the literature endorses.

**What's genuinely missing (confirms DEVELOPMENT-PLAN.md gap #1 and #3, sharpens them):**
- No judge-calibration step anywhere — `/sofi-spec-review`'s Fable "hard gate" acts as an LLM-as-judge but there's no gold-set calibration, no stated escape-hatch ("insufficient evidence"), and no family-diversity rule (Fable judging work that other Claude-family agents produced is exactly the self-enhancement-bias configuration the literature warns about). Concrete fix: add "Unknown/needs human" as a valid `/sofi-spec-review` verdict, and periodically spot-check Fable verdicts against human/CEO review.
- Security squad's gap (adversarial self-verify + per-finding execution plans) maps directly onto the courtroom/fixed-role-debate pattern — right instinct, currently unimplemented. A concrete, cheap version: one pass proposes the fix, a second (different-model or differently-prompted) pass argues why the fix might be wrong/incomplete before it's applied.
- No pass^k-style reliability check anywhere in the gate system — gates check a single pass of `sofi_verify.py`, not whether the same feature/build is *reproducibly* correct across repeated runs. For a 9-gate lifecycle this is a real gap at Gate 5 (Quality) and Gate 6 (Staging/UAT) specifically.
- No confidence-calibration doctrine — agents are never instructed to distinguish "I'm confident" from "I verified this," and nothing stops a spawned agent from self-reporting completion the way Anthropic's harness post explicitly warns against (flight booked ≠ trust the sentence, check the DB row). Worth one line in `thinking-and-work.md`: "self-reported completion is not evidence; the mechanical gate is."

**What would be over-engineering for SOFI right now:** full multi-agent debate infrastructure (SOFI's tier-isolation + RCCF delegation already gives structured, role-fixed handoffs — reinventing free-form debate would be a regression, not a feature); training a dedicated process-reward-model / verifier model (SOFI has no training pipeline and routes through frontier APIs — PRM training is out of scope; the *practice* of process-level checks, not the trained model, is what transfers, and `sofi_verify.py`'s per-layer checks already approximate it); semantic-entropy / token-level hallucination detectors (require internal model access to logprobs/entropy that Claude Code's harness doesn't expose the way a self-hosted eval stack would) — SOFI should stay at the behavioral/outcome-verification layer it already has, not chase internals it can't instrument.
