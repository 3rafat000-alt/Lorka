# Multi-Agent LLM Orchestration Architectures (2025-2026)

## State of the art (2025-2026)

Three dominant topologies compete in production: **centralized orchestrator-worker**, **supervisor** (a constrained orchestrator variant), and **swarm/peer-to-peer handoff**. A 2026 survey (arXiv:2601.13671) frames orchestration as a "control plane" with four functions — planning/policy, execution/control, state/knowledge, quality/ops — and finds enterprise deployments (BFSI, legacy-modernization SWE) overwhelmingly choose centralized control over decentralization because compliance/auditability outweighs the flexibility peer-to-peer buys. Decentralized DAG-based approaches (AgentNet, AgentNet++) exist mainly in research: agents self-organize into clusters and route without predefined workflows, claiming 23% higher task completion and 40% less communication overhead than flat mesh — but no cited production system runs pure P2P at scale.

**Anthropic's multi-agent research system** (anthropic.com/engineering/multi-agent-research-system) is the most-cited real-world case study: a lead agent (Opus) decomposes a query, spawns 3-5 Sonnet subagents in parallel (each with its own context window and 3+ concurrent tool calls), and a separate citation agent handles attribution. It beat single-agent Opus by 90.2% on their internal eval and cut research time up to 90% on complex queries — but burns ~15x the tokens of a single chat turn, and **token usage alone explains 80% of performance variance** (3 factors explain 95%). Failure modes they had to engineer around: agents spawning 50 subagents for trivial queries, vague task boundaries causing duplicate work, subagents preferring SEO content over authoritative sources, and agents not knowing when to stop. Production required full tracing (non-determinism breaks normal debugging) and "rainbow deployments" so mid-flight agents aren't killed by deploys. Their companion piece ("When and how to build multi-agent systems," claude.com/blog) is more conservative: default to the simplest single-agent approach; only fragment when you have (1) context pollution across unrelated subtasks, (2) genuinely parallelizable independent work, or (3) tool/persona specialization conflicts (e.g., an agent juggling 20+ tools). It explicitly warns against splitting by *pipeline phase* (plan/implement/test) — that's sequential, not parallel, and just adds coordination tax without a benefit.

**LangGraph's supervisor vs. swarm benchmark** (langchain.com/blog/benchmarking-multi-agent-architectures) is the sharpest empirical comparison available. Swarm (agents hand off control directly via a `Command`, no return trip through a coordinator) beats supervisor (central node must receive every sub-agent reply, translate/re-phrase it, then respond) on both task score and token cost as distractor-domain count grows — because supervisor pays a "translation tax": sub-agents can't answer the user directly, so the supervisor re-generates their output. LangChain closed ~50% of that gap by (a) stripping handoff scaffolding messages from the context, (b) forwarding sub-agent output verbatim instead of re-synthesizing it, and (c) naming handoff tools so the LLM's routing decision is unambiguous. Their practical verdict: start with a flat supervisor; only add hierarchy tiers once a measured bottleneck justifies the extra coordination cost — hierarchy is not free, each additional level of "ask the parent" adds a full LLM round-trip's worth of latency and tokens.

OpenAI's Agents SDK institutionalizes swarm-style **handoffs**: any agent can expose a `transfer_to_X` tool; a triage agent classifies inbound intent and permanently hands the conversation (with full history) to the specialist — no supervisor round-trip, but also no re-aggregation, so this pattern only fits single-owner-at-a-time workflows (customer support triage), not fan-out/fan-in research tasks.

## Concrete techniques worth adopting

1. **Context-boundary decomposition, not phase decomposition** — split agents along "what needs an isolated context window" (independent research paths, blackbox verification), never along sequential pipeline stages of the *same* work. Sequential splits create coordination overhead with no parallelism payoff (Anthropic).
2. **Verbatim result-forwarding instead of re-synthesis** — when a coordinator receives a worker's output, pass it through unmodified rather than having the coordinator LLM re-write/summarize it. This alone recovered ~50% of the swarm/supervisor performance-and-cost gap (LangChain).
3. **Explicit stop conditions in the delegation contract** — give every spawned agent an objective, output format, tool/source guidance, AND explicit task boundaries (including when to stop), because agents without a stop signal over-search or spawn runaway sub-work (Anthropic's 50-subagent failure mode).
4. **Effort-scaled fan-out** — route trivial tasks to 1 agent / few tool calls, complex tasks to N agents with clearly non-overlapping responsibilities; don't fan out by default. Mirrors SOFI's routing ladder but at the *fan-out width* dimension, not just model tier.
5. **Full production tracing for non-deterministic multi-agent runs** — standard step-through debugging breaks when subagent behavior varies run-to-run; you need persisted traces of every delegation + tool call to diagnose failures after the fact (Anthropic).
6. **LLM-as-judge with a fixed rubric for agent-output QA** — a single LLM call scoring 0.0-1.0 + pass/fail across named criteria (factual accuracy, citation accuracy, completeness, source quality, tool efficiency) tracked closest to human judgment, cheaper than multi-judge ensembles (Anthropic).
7. **Coordination-cost accounting before adopting hierarchy** — model token/latency cost per added tier (each hop = one more LLM round trip, ~200 tokens routing + full worker call) and only add a tier when a *measured* bottleneck justifies it, not speculatively (LangChain, arXiv:2601.13671 cost note: workflows that cost $0.50 in testing can hit $50k/month at 100k executions once orchestrator overhead compounds).

## How it applies to SOFI

**Already right:** SOFI's tier-Advisor structure is functionally the orchestrator-worker / supervisor pattern the literature converges on for anything needing auditability — cross-tier work must route Advisor→Advisor, which is exactly the "control plane enforces policy alignment across peer specialists" model arXiv:2601.13671 found dominant in regulated/enterprise deployments (SOFI's gate lifecycle + git-backed brain is effectively that paper's "state & knowledge management" + "quality & ops management" functions, made concrete). The RCCF delegation block already encodes Anthropic's #3 finding (objective + format + tool guidance + boundaries) almost verbatim — good validation, not a gap. The economic routing ladder (haiku→sonnet→fable→opus) is the right instinct on the *model* axis; the literature's "effort-scaled fan-out" (#4) is the same instinct applied to *fan-out width*, which SOFI does not currently model at all.

**Genuinely missing, worth adopting:**
- **No stated context-boundary rule for when to spawn parallel specialists vs. sequential handoff.** SOFI's gate lifecycle is inherently sequential (0→8), so most delegation today is naturally handoff-style, not fan-out. But `/sofi-feature` and `/sofi-audit` already do internal parallel-ish sweeps (Python scan → multiple specialist fixes) — worth explicitly codifying "spawn parallel only when subtasks are context-independent" so squads don't reflexively fan out RCCF blocks for sequential work and pay the coordination tax for nothing.
- **No verbatim-forwarding discipline in HANDOFFS.md.** If an Advisor re-summarizes a specialist's findings before writing the ticket, that's the exact "translation tax" LangChain measured costing real tokens and losing fidelity. Worth a one-line rule: paste specialist findings into HANDOFFS.md verbatim (with file:line), don't re-narrate them.
- **No fan-out cost accounting in routing.yaml.** The ladder prices model tier but not squad *width* (how many agents spawned per gate). A cheap addition: routing.yaml could note expected token multiplier per parallel squad size, so CEO/Advisors can justify a 3-agent parallel sweep the way Anthropic justifies "value of task high enough to pay for the increase."

**Would be over-engineering for SOFI right now:**
- **True peer-to-peer / decentralized mesh (AgentNet-style).** SOFI's strict tier-isolation + gate model is a deliberate compliance/auditability choice (git-law, no cross-tier bleed) — exactly the enterprise pattern the survey says trades autonomy for governance. Adding P2P handoffs between specialists would break `tickets.validate_tier_boundary()` and buys speed SOFI doesn't need (it's not doing live multi-turn chat triage; it's a gated SDLC).
- **OpenAI-style permanent handoffs (transfer_to_X with full context takeover).** Fits customer-support triage, not a 9-gate SDLC where the brain (STATE/CONTEXT/DECISIONS/HANDOFFS) must survive independent of any single agent's context — SOFI's git-backed brain is already a *better* fit for its problem than either swarm or handoff-SDK patterns, which assume in-memory conversational state.
- **50+ subagent parallel fan-out.** SOFI's per-role tooling and tier isolation already bound blast radius; adopting Anthropic's aggressive 10+-subagent parallelism would multiply token cost (15x baseline) for a domain (software SDLC, which Anthropic explicitly flags as *poorly suited* to heavy parallelization — "coding work has fewer parallelizable components") where it likely won't pay off.
