# Designing Agent System Prompts and Role Specs (2026)

## State of the art (2025-2026)

**Orchestrator-worker is the dominant multi-agent pattern.** Anthropic's production Research system (lead agent + parallel subagents) beat single-agent by 90.2% on internal evals but burns ~15x the tokens of a single chat call — the frontier explicitly trades cost for reliability, and only when the task is genuinely parallelizable/high-value. Anthropic's own guidance ("When to use multi-agent systems") is to default to single-agent and only fan out when subtasks are truly independent.

**The four-part subagent contract is the load-bearing artifact.** Anthropic's engineering blog states plainly: every subagent needs (1) an objective, (2) an output format, (3) tool/source guidance, (4) explicit task boundaries — "miss any of the four and the subagent drifts." Their failure-mode log is concrete: vague missions ("research the semiconductor shortage") caused duplicate work across parallel agents; the fix was per-agent scoped decomposition plus explicit **effort-scaling rules** hardcoded into the prompt (1 agent/3-10 calls for simple fact-finding, 10+ agents for complex research) — the orchestrator is told directly how much autonomy each task class deserves rather than left to infer it.

**Bounded autonomy is now a named discipline with empirical backing, not just a slogan.** The 2026 arXiv line (typed action contracts, consumer-side execution enforcement) and multiple practitioner writeups converge on: constraints improve task completion, not just safety. One controlled study cited across sources: unconstrained agents with more actions and fewer validation checks completed *fewer* tasks (17/25 vs 23/25) than constrained ones and produced unsafe mutations — more freedom did not produce more utility. "Flow engineering" (explicit control-flow/state-transition design around LLM calls, treated as software architecture) and "budgeted autonomy" (hard limits on tokens/time/tool-calls/cost with fail-safe termination) are the concrete mechanisms.

**Tool/schema design is prompt engineering, not just typing.** Anthropic's "Writing Effective Tools for AI Agents" (evaluation-driven, iterative) reports that refining tool *descriptions* alone (unambiguous names like `user_id` not `user`, namespacing like `asana_projects_search`, a `response_format: concise|detailed` enum, actionable prompt-engineered error messages) produced measurable gains — one case a 40% reduction in downstream task completion time from description edits alone, no code change.

**Spec-driven development (SDD) crossed into the mainstream in 2025-2026.** Every major agentic tool (GitHub Spec Kit, AWS Kiro, Claude Code, Cursor, Google Antigravity) shipped a spec-first mode. The arXiv framework defines a maturity ladder: Spec-First (spec guides but drifts) → Spec-Anchored (spec + code kept aligned by automated checks) → Spec-as-Source (humans edit only the spec; code is regenerated). The mechanism that makes a spec "executable" is enforcement — BDD/Gherkin/OpenAPI turned into failing tests when implementation diverges — not just a written doc. Claude Code's own docs now formalize this as "interview → write SPEC.md → fresh session implements" specifically because a spec written before implementation starts prevents scope drift better than correcting mid-session.

**Claude Code subagent design orthodoxy (2026):** subagents get their *own* system prompt + tool allowlist + permission mode (they do NOT inherit the parent's full prompt); tool access should be minimal-necessary (Read/Glob/Grep-only agents are cheaper, more trustworthy, easier to debug); the `description` field should be action-oriented, not capability-vague, because it's what routes delegation; and an **adversarial review subagent** (fresh context, sees only the diff + criteria, not the reasoning that produced it) is the recommended check before marking work done — explicitly to avoid the implementer grading its own work.

## Concrete techniques worth adopting

1. **Four-part contract enforcement, machine-checked** — every delegation carries objective + output format + tool/source scope + explicit boundaries; a linter/hook rejects delegations missing any part. *Why:* Anthropic's own root-cause for subagent drift is missing exactly one of these four fields.
2. **Effort-scaling table in the orchestrator's prompt, not inferred** — hardcode task-class → agent-count/call-budget mappings (e.g., "single-file fix: 1 agent, no subagents; cross-tier feature: N specialists"). *Why:* prevents both under- and over-spawning; Anthropic saw both failure directions until they hardcoded the table.
3. **Budgeted autonomy / fail-safe termination** — explicit hard caps on tool calls, tokens, or wall-clock per delegated unit, with a defined stop behavior on breach (not "keep trying"). *Why:* empirically, unconstrained agents complete *fewer* tasks and produce more unsafe side-effects than bounded ones.
4. **Adversarial self-verify subagent, fresh-context** — a second agent reviews only the diff/output + the original criteria, never the reasoning trace, before work is marked complete. *Why:* removes the implementer-grades-itself blind spot; this is now bundled as Claude Code's `/code-review` pattern.
5. **Tool description as prompt-engineering artifact, iteratively evaluated** — unambiguous param names, `response_format` verbosity control, actionable (not opaque) error strings, namespacing by service+resource. *Why:* measured 40% completion-time improvement from description-only edits, zero code change — the cheapest lever available.
6. **Spec-as-contract with enforcement, not just a written brief** — a frozen spec becomes "executable" only when something (test, schema validator, gate check) fails loudly on divergence. *Why:* the whole 2025-2026 SDD literature's finding is that specs without enforcement drift exactly like code-first docs did.
7. **Output abstraction / handoff-by-reference** — subagents write large artifacts to storage and pass back a lightweight pointer/summary rather than inlining full output through the orchestrator. *Why:* Anthropic reports this improves both fidelity (no lossy re-summarization) and token cost on multi-hop handoffs.
8. **State checkpointing before context exhaustion** — agents persist plan/progress to external memory proactively, not only at natural stopping points. *Why:* Anthropic's lead agent does this specifically to survive token-budget limits without losing the plan.

## How it applies to SOFI

**Already right:** SOFI's RCCF block *is* essentially Anthropic's four-part contract (Role/Context ≈ tool+source scope, Command ≈ objective+boundaries, Format ≈ output format) — this is not a gap, it's convergent design with the frontier pattern, and SOFI enforces it structurally via `/sofi-delegate` rather than hoping agents remember it. The tiered roster + Advisor-only cross-tier routing is a stronger, more explicit form of "task boundaries" than most multi-agent frameworks bother with. The routing ladder (haiku→sonnet→fable→opus) is SOFI's own version of effort-scaling-by-task-class, already hardcoded rather than inferred — matches technique #2 directly. The 9-gate lifecycle with exit bars is a real enforcement mechanism for specs (STATE/CONTEXT/DECISIONS/HANDOFFS as the spec-anchored brain), closer to "Spec-Anchored" maturity than most teams reach.

**Genuinely missing, worth adopting:**
- **Budgeted autonomy (#3) is absent.** Nothing in `engine/routing/routing.yaml` or the delegation protocol caps tool-calls/tokens/wall-clock per spawned agent with a defined fail-safe stop. Given the DEVELOPMENT-PLAN gap #3 (security squad lacks adversarial self-verify + per-finding execution plans) and gap #4 (RCCF lacks frozen-brief enforcement), this is the same root cause showing up twice: no hard resource/scope ceiling on a delegation, only soft prose instructions.
- **Adversarial self-verify (#4) is only partial.** `/code-review` exists generically but SOFI's own gate-bars don't systematically spawn a fresh-context reviewer against the *original ticket criteria* (as opposed to a diff-only code review) before a gate advances — this is exactly DEVELOPMENT-PLAN gap #3, now corroborated by frontier practice, not just an internal hunch.
- **Frozen-brief + clarifying-question branch (DEVELOPMENT-PLAN gap #4)** maps directly to Claude Code's own now-canonical "interview → write SPEC.md → fresh session implements" pattern. This is not over-engineering — it's the industry's answer to the exact failure Anthropic and the SDD papers both name (agents solving the wrong problem because scope was inferred, not fixed).
- **Structured frontmatter on brain files (gap #6)** matches the SDD "machine-readable spec" principle (OpenAPI/Gherkin-style structure over prose) — low-cost, directly reduces query/parsing cost for `sofi_tools`, worth doing.
- **Tool-description iteration (#5) is not a discipline SOFI has** — `engine/tooling/registry.yaml` entries and MCP/CLI tool descriptions used by agents haven't been evaluation-tuned the way Anthropic prescribes. Cheap to pilot on one high-traffic tool (e.g., `sofi checkpoint` or `sofi gate-check`) before generalizing.

**Would be over-engineering for SOFI right now:**
- Full async/parallel subagent coordination (Anthropic's own noted trade-off — SOFI's tier isolation + ticket-based sequencing already sidesteps the cross-subagent race conditions this solves, and SOFI doesn't have Anthropic's research-fanout workload shape).
- Spec-as-Source (full code regeneration from spec) — SOFI's codebase isn't at the maturity/tooling level to regenerate rather than incrementally patch; Spec-Anchored (current trajectory) is the right target, not the ceiling.
- A dedicated typed-action-contract enforcement layer (the arXiv "consumer-side execution" pattern) — that's infrastructure-grade for enterprises running many untrusted agents; SOFI's guard hooks + tier-boundary validation already cover the equivalent surface at much lower cost.
